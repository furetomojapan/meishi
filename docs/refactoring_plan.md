# デジタル名刺サイト 完全リファクタリング計画書

作成: 2026-06-11（v5.10 / GAS v3.1 時点）

## 0. 基本方針

このリファクタリングは「**動いているサイトを一度も壊さない**」ことを最優先にする。そのために次の原則を全フェーズで守る。

1. **各フェーズは単独でデプロイ可能**にし、フェーズ末に必ず本番で動作確認してから次へ進む
2. **挙動を変えるフェーズと変えないフェーズを混ぜない**（掃除と機能変更を同じコミットにしない）
3. フェーズ開始前にタグ（`git tag pre-phaseN`）を打ち、いつでも戻れるようにする
4. GASの変更を含むフェーズは、必ずスプレッドシートの**バックアップコピー**を取ってから行う
5. 1フェーズ = 1つのまとまったコミット群。session_log.md に毎フェーズ記録する

## 1. 現状の棚卸し（実測値）

### 1-1. フロントエンド（index.html / 約2,550行・208KB）

| 問題 | 実測 | 影響 |
|---|---|---|
| 単一巨大ファイル | 2,550行・208KB | 変更のたびに全体を壊すリスク。差分レビュー困難 |
| Babel standalone をブラウザ内で実行 | CDN読込 | 初回表示が遅い（毎回ブラウザ内でJSXコンパイル）。本番非推奨の構成 |
| App() 1コンポーネントに全機能 | useState 64個 | 閲覧・編集・管理画面・タグ・PIN認証が密結合。1つ触ると全部に波及 |
| GAS_READY 恒真分岐 | 21箇所 | GAS導入前のフォールバックコード。今は絶対に通らない死に分岐 |
| 死にコード | `/api/names` fetch（存在しないAPI）、GitHubリポジトリ一覧からの名前推測、配列形式の旧データ対応 | 読み手を混乱させる。通信の無駄 |
| ユーザー編集と管理者編集の二重実装 | プロフィール入力欄・デザインタブがほぼコピペで2系統 | 修正漏れの温床（実際に「管理画面も同時変更」ルールが必要になっている原因） |
| normalizeProfile系の重複 | defaultProfile が3箇所に別々に定義 | デフォルト値の不整合リスク |
| インラインstyleとTailwindの混在 | カード描画部 | デザイン変更時の調査コスト |

### 1-2. バックエンド（gas_backend.js / 約1,000行）

| 問題 | 実測 | 影響 |
|---|---|---|
| 巨大switch文のdoPost | 約25アクション直書き | 認証チェックの書き忘れがそのまま脆弱性になる構造 |
| シート全読みの乱発 | getDataRange().getValues() が17箇所 | 1リクエスト中に同じシートを何度も全読み。ユーザー増で確実に遅くなる |
| 書き込みロックなし | LockService 未使用 | 同時保存で行のズレ・上書き事故の可能性 |
| 旧互換エイリアス | `delete_user`（= admin_delete_user の複製） | 攻撃面の無駄な拡大 |
| PINが平文保存 | pin列 | 管理者がPIN閲覧できる仕様とのトレードオフ（要判断） |
| テストなし・手動デプロイ | — | 「貼り替え→再デプロイ→initSheets」の手作業はミスりやすい |

### 1-3. リポジトリ・資産

| 問題 | 実測 | 影響 |
|---|---|---|
| 画像が重い | PNG合計 約9.5MB（最大2.0MB/枚） | 名刺1枚見るだけのサイトとして過大。スマホで遅い |
| 重複ファイル | `image1_ichiro_suzuki copy.png` 等の copy 2枚（約3MB） | 完全な無駄 |
| ルート直下に全部置き | 画像・PDF・docs・コードが混在 | 構造が読めない |
| session_log.md がルートとdocsに二重存在 | 2ファイル | どちらが正か不明になる |
| リポジトリ肥大 | .git 15MB / 139コミット | 画像をgitで持ち続ける運用の限界が近い |

### フェーズ5: GASバックエンドの再構築

**目的**: 「読みやすく・速く・安全に壊れない」サーバーにする。

- [ ] **clasp 導入**: ローカルのコードを `clasp push` でGASへ反映（手動コピペ廃止）。ファイル分割が可能になる
  - `http.gs`（doGet/doPost ルーティング）/ `auth.gs` / `users.gs` / `tags.gs` / `licenses.gs` / `util.gs`
- [ ] **ルーティングテーブル化**: switch文 → `{ action: { auth: "session"|"admin"|"none", handler } }` のマップに。認証の書き忘れが構造的に起きなくする
- [ ] **シート読み取りの一本化**: リクエスト先頭で usersテーブルを1回だけ読み、以降は渡し回す（getDataRange 17回 → 1〜2回）
- [ ] **LockService**: 全書き込み系アクションに `LockService.getScriptLock()` を導入（同時保存の競合防止）
- [ ] **入力検証層**: name/pin/token/tags の形式チェックを共通関数に
- [ ] **エラーレスポンス統一**: `{ success:false, code:"AUTH_FAILED", error:"…" }` の形に統一（フロントの分岐が安定する）
- [ ] **判断ポイント**: PINのハッシュ化。現仕様は「管理者がPINを閲覧できる」ことが前提（admin_get_all_pins）。ハッシュ化すると閲覧不可になり「リセットのみ」運用になる。セキュリティ優先ならハッシュ化推奨
- [ ] sessions シートの定期掃除をトリガー（時間主導型）に移し、リクエスト時の掃除を軽量化

**受け入れ基準**: 全アクションの動作確認（スモークテスト+管理操作一式）、保存系の同時実行テスト、応答時間が現状同等以上
**リスク**: 高（全アクションに触る）。対策: フェーズ0で作ったテスト用デプロイURLで全確認後に本番URLへ反映。シートバックアップ必須
**目安**: 1日

## 付録A: スモークテスト チェックリスト（毎フェーズ末に実施）

1. 名刺URL（フル）で開く → カード表示・フリップ・リンクタップ
2. タグ用URL（zt〜）で開く → 限定表示注記・電話/住所がタグ用値 or 非表示・編集ボタン非表示
3. 名刺を編集 → PIN認証（成功/失敗/5回失敗ロック）→ 保存 → 再読込で反映確認
4. タグタブ → 可視人数表示・全体公開トグル・保存（24時間制限の表示）
5. タグ仲間 → 一覧表示 → 他人の名刺が限定URLで開く
6. 画像ダウンロード・シェア・QR
7. 管理画面 → ログイン（初期パスなら強制変更）→ ユーザー作成/編集/PRO切替/PINリセット/タグ編集
8. 新規登録（welcome.html）→ メール受信 → 初回PIN設定
9. ブラウザコンソールにエラーが出ていないこと

## 付録B: ロールバック手順

- フロント: `git revert` または `git reset --hard pre-phaseN` → push（Pagesが旧版に戻る）
- GAS: デプロイ管理 → 旧バージョンを選んで再デプロイ（コード貼り替え不要）
- データ: バックアップシートから該当シートをコピーで戻す
